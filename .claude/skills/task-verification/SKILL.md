# タスク検証スキル

タスクファイル（`docs/タスク/` 等）に記載された要件と、実際の実装コードを突き合わせて、
仕様適合性を検証する。

## 発動タイミング

- ユーザーが「タスクの実装が合っているか確認して」「仕様通りか検証して」と指示した場合
- タスクの実装が完了し、✅ を付ける前の最終確認として
- コードレビュー前の自己チェックとして

## 前提

- タスクファイルが存在する（デフォルト: `docs/タスク/` 配下）
- タスクファイルに設計書へのリンクがある場合、設計書も参照する
- 対象のコードがリポジトリ内にある

---

## 検証フロー

### Step 1: タスクファイルを読む

対象タスクファイルから以下を抽出する:

- **セクション一覧**（`##`）: 機能の大分類
- **サブタスク一覧**（`###`）: 個別の実装項目
- **各サブタスクの要件**: 箇条書きの実装内容
- **検証セクション**: チェックリスト項目

```
入力: docs/タスク/XX_{機能名}.md
出力: 構造化された要件リスト
```

### Step 2: 設計書を読む（リンクがある場合）

タスクファイル冒頭の設計書リンクを辿り、より詳細な仕様を確認する。
タスクの箇条書きだけでは判断できない要件（データ構造、API レスポンス形式、エッジケース等）を補完する。

### Step 3: 実装コードを特定する

サブタスクの記述から対象ファイル・関数を特定する:

1. タスクに明示的なファイルパスがあればそれを使う
2. なければ、タスクの内容からキーワードを抽出してコードベースを検索する
3. 関連するテストファイルも併せて特定する

### Step 4: 要件ごとに突合する

各サブタスクの要件を1つずつ、実装コードと突き合わせる。

**チェック観点:**

| 観点 | 確認内容 |
|------|---------|
| **存在性** | 要件に対応するコード（関数・クラス・エンドポイント等）が存在するか |
| **正確性** | 実装が仕様通りの動作をするか（ロジック、データ構造、戻り値） |
| **完全性** | 要件の全項目がカバーされているか（一部だけ実装されていないか） |
| **整合性** | 設計書の制約（バリデーション、上限値、エラー処理等）が守られているか |
| **副作用** | 仕様にない余計な変更や、既存機能への影響がないか |

### Step 5: 検証レポートを出力する

---

## 検証レポートフォーマット

```markdown
# タスク検証レポート: XX. {機能名}

検証日時: YYYY-MM-DD
対象タスク: docs/タスク/XX_{機能名}.md

## サマリ

| 項目 | 結果 |
|------|------|
| サブタスク総数 | N |
| 適合 | N |
| 不適合 | N |
| 要確認 | N |
| 適合率 | XX% |

## 詳細

### XX.1.1 {サブタスク名} — ✅ 適合

- [x] 要件A: {対応するコードの場所}
- [x] 要件B: {対応するコードの場所}

### XX.1.2 {サブタスク名} — ❌ 不適合

- [x] 要件A: {対応するコードの場所}
- [ ] 要件B: **未実装** — {何が足りないか}
- [ ] 要件C: **仕様と乖離** — {期待値 vs 実際の動作}

### XX.2.1 {サブタスク名} — ⚠️ 要確認

- [x] 要件A: {対応するコードの場所}
- [?] 要件B: コードは存在するが、エッジケースの動作が不明

## 検証セクション突合

タスクファイルの検証チェックリストとの対応:

- [x] 検証項目1 — テスト/コードで確認済み
- [ ] 検証項目2 — **未検証**

## 指摘事項

1. **[不適合]** XX.1.2 の要件B: {具体的な修正内容}
2. **[要確認]** XX.2.1 の要件B: {確認が必要な内容}
3. **[推奨]** {仕様外だが改善が望ましい点}
```

---

## 判定基準

### ✅ 適合

- 要件に対応するコードが存在する
- ロジックが仕様通りである
- バリデーション・エラー処理が設計書の記述に合致する

### ❌ 不適合

- 要件に対応するコードが存在しない（**未実装**）
- コードは存在するが仕様と異なる動作をする（**仕様乖離**）
- 設計書で定められた制約が守られていない（**制約違反**）

### ⚠️ 要確認

- コードは存在するが、静的解析だけでは正確性を判断できない
- テストがないため動作確認が必要
- 仕様が曖昧で、解釈が分かれる部分

---

## 使い方の例

### 単一タスクの検証

```
ユーザー: タスク08の実装が合っているか確認して
```

→ `docs/タスク/08_{機能名}.md` を読み、全サブタスクを検証

### 特定セクションの検証

```
ユーザー: タスク08のセクション2だけ確認して
```

→ `## 08.2` のサブタスクのみを対象に検証

### 完了前チェック

```
ユーザー: タスク08を完了にしていいか確認して
```

→ 全サブタスク + 検証セクションを突合し、✅ / 【完了】を付けてよいか判断

---

## 注意事項

- このスキルは**静的な突合チェック**であり、実行時の動作テストは含まない
- 動作テストが必要な場合は、プロジェクトのテストスキル（E2E テスト等）と組み合わせる
- LLM 推論に依存する判定（「自然な応答か」等）は ⚠️ 要確認 として報告する
- 検証レポートはタスクファイルには書き込まず、ユーザーへの出力として提示する
