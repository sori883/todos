# タスク管理スキル

マークダウンベースのタスクファイルで開発ライフサイクルを管理する。
設計→タスク生成→実装→テスト→完了の一気通貫ワークフロー。

## 前提

- 設計書ディレクトリ（デフォルト: `docs/設計/`）に設計書がある
- タスクディレクトリ（デフォルト: `docs/タスク/`）にタスクファイルを格納する
- ディレクトリ名はプロジェクトに合わせて変更可（`docs/design/`, `docs/tasks/` 等）

## ディレクトリ構造

```
docs/
├── 設計/                    # 設計書（カテゴリ別サブディレクトリ）
│   ├── {カテゴリA}/
│   │   ├── README.md
│   │   └── {機能名}.md
│   └── {カテゴリB}/
│       └── {機能名}.md
├── タスク/                  # タスクファイル（連番管理）
│   ├── 01_{機能名}【完了】.md
│   ├── 02_{機能名}【完了】.md
│   └── 03_{機能名}.md      # 未完了
└── テスト/                  # テスト関連ドキュメント（任意）
    └── テストガイド.md
```

## Phase 1: 設計書からタスクを生成する

ユーザーが「この設計書からタスクを作って」と指示した場合に実行する。

### 手順

1. **設計書を読む**

対象の設計書の内容を確認する。

2. **既存タスクの連番を確認**

タスクディレクトリの既存ファイルから最大の連番を確認し、次の番号を決める。

3. **タスクファイルを生成**

設計書の各セクションをタスクのセクション（`##`）に、
各要件をサブタスク（`###`）にマッピングする。

**マッピングルール:**

| 設計書の要素 | タスクファイルの要素 |
|-------------|-------------------|
| 大セクション（## 概要 等） | セクション見出し（## XX.1） |
| 要件・仕様の箇条書き | サブタスク（### XX.1.1） |
| データ構造・スキーマ | DB / データモデルタスク |
| API / インターフェース定義 | API / ツール定義タスク |
| UI / フロント要件 | フロントエンドタスク |
| インフラ / デプロイ要件 | インフラタスク |

4. **検証セクションを追加**

設計書の要件から検証項目を抽出し、チェックリストとして追加する。

5. **タスクファイルに設計書へのリンクを含める**

### タスクファイルテンプレート

```markdown
# {連番}. {機能名}

{1〜2行の概要説明}

設計: [docs/設計/{カテゴリ}/{ファイル名}.md](../設計/{カテゴリ}/{ファイル名}.md)

## {連番}.1 セクション名

### {連番}.1.1 タスク名

- 実装内容（箇条書き）

### {連番}.1.2 タスク名

- 実装内容（箇条書き）

---

## {連番}.N 検証

- [ ] 検証項目1
- [ ] 検証項目2
```

## Phase 2: タスクを参照しながら実装する

ユーザーが「タスク XX を実装して」と指示した場合に実行する。

### 手順

1. **タスクファイルを読む**

未完了タスク（ファイル名に `【完了】` がないもの）を確認し、対象タスクを読む。

2. **設計書を参照する**

タスクファイル冒頭のリンクから設計書を読み、詳細仕様を確認する。

3. **未完了のサブタスク（✅ がないもの）を特定する**

`###` 見出しに ✅ がないものが未完了。上から順に実装する。

4. **実装する**

プロジェクトの開発フローに従う:
- 非自明なタスクは計画を立ててから実装する
- 必要に応じてコードレビュー・セキュリティレビューを行う

5. **サブタスク完了時にマークする**

実装が完了したサブタスクに ✅ を付け、実装内容を箇条書きで記録する:

```markdown
### XX.1.1 タスク名 ✅

- 実装したファイルやモジュール
- 主要な実装内容の要約
```

6. **セクション完了時にマークする**

セクション配下の全サブタスクが完了したら、セクション見出しにも ✅:

```markdown
## XX.1 セクション名 ✅
```

---

## Phase 3: テストを実行する

タスクの全サブタスクが完了したら、検証セクションを実行する。

### 手順

1. **検証セクションを確認**

タスクファイル末尾の `## XX.N 検証` セクションのチェックリストを確認する。

2. **テストを作成・実行**

検証項目をカバーするテストを作成して実行する。
プロジェクトのテストフレームワーク・テストスキルがあればそれに従う。

3. **検証項目を更新**

テスト結果に応じてチェックリストを更新:

```markdown
- [x] 検証に通った項目
- [ ] まだ通っていない項目
```

4. **失敗した場合**

テスト失敗時は対象のサブタスクに戻って修正し、再テストする。

---

## Phase 4: タスクを完了にする

全サブタスクと検証が完了したら、タスクファイルを完了状態にする。

### 手順

1. **全 ✅ と全 [x] を確認**

- 全ての `###` 見出しに ✅ が付いているか
- 全ての `##` 見出しに ✅ が付いているか
- 検証セクションの全チェックボックスが `[x]` か

2. **ファイル名に【完了】を付与**

```bash
mv "docs/タスク/XX_機能名.md" "docs/タスク/XX_機能名【完了】.md"
```

---

## ファイル命名規則

```
{連番}_{機能名}{ステータス}.md
```

| 要素 | 形式 | 例 |
|------|------|------|
| 連番 | 2桁ゼロ埋め | `01`, `08`, `12` |
| 機能名 | 簡潔な名前 | `認証`, `API設計`, `user_auth` |
| ステータス | 全タスク完了時のみ `【完了】` を付与 | `【完了】` |

## 完了マークのルール

| 対象 | マーク方法 | タイミング |
|------|-----------|-----------|
| サブタスク（`###`） | 見出しに ✅ を追加 | 実装完了時 |
| セクション（`##`） | 見出しに ✅ を追加 | 配下の全サブタスク完了時 |
| 検証項目 | `- [ ]` → `- [x]` | テスト通過時 |
| ファイル名 | `【完了】` を付与 | 全セクション・全検証完了時 |

## クイックリファレンス

```bash
# タスク一覧の確認
ls docs/タスク/

# 未完了タスクのみ
ls docs/タスク/ | grep -v 【完了】

# 設計書一覧
find docs/設計 -name "*.md"

# 完了マーク
mv "docs/タスク/XX_機能名.md" "docs/タスク/XX_機能名【完了】.md"

# 再開（完了を取り消し）
mv "docs/タスク/XX_機能名【完了】.md" "docs/タスク/XX_機能名.md"
```
